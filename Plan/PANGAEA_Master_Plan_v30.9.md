## 랜딩페이지 (pangaea.app)

### 목표

방문 15초 안에: (1) "이건 뭐지?" 호기심 → (2) "이런 거구나" 이해 → (3) "써보고 싶다" 전환.

- 프레임워크: Next.js (SSG) + WebGL 히어로.

- SEO: 메타 태그, OG 이미지(바다+섬 스크린샷), 구조화 데이터.
- 다국어: 한국어(기본) + 영어. Phase 2+에서 추가.

- 타이포: Playfair Display (영문 제목) + Noto Serif KR (한글 제목) + IBM Plex Sans (본문).

## UI 디자인 시스템

### DS.1 디자인 방향: Liquid Glass

**핵심 비주얼 언어: 리퀴드 글래스(Liquid Glass)**

바다 위의 유리 — 투명하고, 빛을 머금고, 물결처럼 부드럽다. 판게아의 모든 UI 표면은 "바다 위에 떠 있는 유리판" 느낌.

#### 리퀴드 글래스의 특성

| 특성 | 표현 | CSS/렌더 |
|------|------|---------|
| **반투명** | 배경이 비친다. 바다/블록이 UI 뒤로 살짝 보인다. | `backdrop-filter: blur(16px) saturate(1.2)` |
| **유리 테두리** | 얇은 빛 테두리. 빛 방향에 따라 상단이 밝고 하단이 어둡다. | `border: 1px solid rgba(255,255,255,0.15)` + 상단 하이라이트 |
| **내부 발광** | 선택/활성 상태에서 은은한 내부 글로우. | `box-shadow: inset 0 0 20px rgba(255,255,255,0.05)` |
| **깊이감** | 글래스 레이어 간 미세한 그림자. 떠 있는 느낌. | `box-shadow: 0 8px 32px rgba(0,0,0,0.12)` |
| **물결 곡률** | 모서리는 부드럽게 둥글다. 날카로운 직각 없음. | `border-radius: 12~16px` |
| **색상 틴트** | 글래스에 미세한 색조. 낮=푸른 틴트, 밤=남색 틴트. | `background: rgba(tint, 0.6~0.8)` |

#### 적용 안 하는 것

- ❌ 완전 불투명 카드 (배경이 안 비치면 글래스가 아님)
- ❌ 과한 blur (blur 40px+ → 뭉개져서 정보 전달 방해)
- ❌ 네온 글로우 (사이버펑크 아님 — 은은한 자연광)
- ❌ 흰 배경에 글래스 (바다색 배경이 있어야 글래스가 살아남)

### DS.4 글래스 컴포넌트 라이브러리

#### 기본 글래스 표면

```css
.glass {
  background: rgba(255, 255, 255, 0.08);      /* 낮 모드 */
  backdrop-filter: blur(16px) saturate(1.2);
  border: 1px solid rgba(255, 255, 255, 0.15);
  border-radius: 12px;
  box-shadow:
    0 8px 32px rgba(0, 0, 0, 0.12),           /* 외부 그림자 */

.glass--night {

  box-shadow:
    0 8px 32px rgba(0, 0, 0, 0.3),
```

### DS.5 화면별 UI 구성

#### 대륙 뷰 (메인 화면)

```
┌──────────────────────────────────────────────────────┐
│ ┌ 상단 바 (glass) ─────────────────────────────────┐ │
│ │ PANGAEA  [봉화][지형][깃발][항로][기후]  ⇅ 👁 ☀ ⚙ │ │
│ └──────────────────────────────────────────────────┘ │
│                                                      │
│              [ 바다 + 섬 렌더 영역 ]                  │
│              (전체 화면, 글래스 없음)                  │
│                                                      │
│                                         ┌ 줌 (glass)│
│                                         │ [+]       │
│                                         │ [⊡]       │
│                                         │ [−]       │
│                                         │ 85%       │
│                                         └───────────│
│                                                      │
│ ┌ 범례 (glass, 하단) ─────────────────────────────┐ │
│ │ 조용 ▬▬▬▬ 활발   1칸=1판 · 높이=용량 · 색=활성도 │ │
│ └──────────────────────────────────────────────────┘ │
│ ┌ 하단 탭 바 (glass) ─────────────────────────────┐ │
│ │  🗺 대륙    📝 판    ⚓ 부두    🗿 모아이   👤    │ │
│ └──────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
```

- 바다 렌더는 글래스 뒤로 비침 → 상단 바/하단 바에 바다색이 투영.
- 상단 바 높이: 48px. 하단 탭 바: 56px (모바일) / 좌측 사이드바 60px (데스크탑).

#### 대륙 뷰 + 상세 패널

- 상세 패널: 우측 슬라이드 인. 폭 240px (데스크탑) / 풀 바텀 시트 (모바일).
- 패널 내용: 판 제목 ★ / 토지 › 영역 / 활성도% / 용량KB / 상태 / 유형 / [편집][이동][모아이][삭제].
- 선택된 블록은 발광.

#### 판 목록 뷰

- 상단 바: 📝 판 + 🔍 검색 + 필터 ▾ + 정렬 ▾.
- 토지 탭 (glass pills): [전체] [일터] [배움] [건강] [취미] [부두] ...
- 판 카드 (glass): 토템아이콘 + 제목 + 룬(status·type·priority) + 토지›영역 + 용량 + 시간. 탭하면 에디터 진입.
- [+ 새 판] FAB. 배경: 바다 그라데이션 (대륙 뷰와 동일 배경 유지).

#### 에디터 뷰

- 에디터 상단 바 (glass): ← 대륙 / 제목 / 토지›영역 / [읽기][소스] ⋯
- 메타 패널 (glass, 접이식): status ▾ / type ▾ / priority ▾ / tags / remind.
- 에디터 본문 영역: 반투명이 아닌 **불투명 배경** (가독성 최우선). 낮 `rgba(255,255,255,0.96)` / 밤 `rgba(15,20,35,0.96)`.
- 모바일: 플로팅 툴바 (glass): H B I ≡ ☑ 🔗 📎 / 🗿

#### 모아이 탭

- 상단 바: 🗿 모아이 + 잉크 잔량 (247/300).
- 측량사 리포트 카드 (glass, 골드 테두리): 부두 배정 제안 / 중복 판 / 혼잡 경고 + [승인][무시][상세보기].
- 사용자 말풍선: 우측 정렬, 액센트 틴트 글래스. 모아이 말풍선: 좌측 정렬, 기본 글래스. 응답에 (스킬명, N Ink) 표시 + [승인][수정][취소].
- 하단 입력 바 (glass).

## 0) 한 줄 정의

**개인의 기록이 중구난방으로 흩어지는 문제**를, **토지(Land) 기반 지도(대륙) 시각화**와 **모아이의 AI 정리(석공 + 측량사)**로 해결하는 **로컬-퍼스트 파일 기반 지식 서비스**.

Phase 1: 마크다운 중심 노트 → Phase 2: 다양한 파일 형식 지원 → Phase 3+: **앱 실행이 가능한 개인 지식 OS**로 진화한다.

---

## 1) 제품 DNA(원칙)

**1차 고객:** 개인(혼자 쓰는 기록). Phase 2+에서 커뮤니티로 확장.

1. **파일이 기본이다**: 판의 기본은 `.md`이지만, 어떤 확장자든 판이 될 수 있다. 판게아는 파일을 다루는 서비스다.
2. **속도가 신뢰다**: 입력·렌더·검색이 즉시 반응(로컬-퍼스트).
3. **시각화가 목표다**: 유저는 '대륙 성장'을 목격하고 성취를 느낀다.
4. **가독성 우선**: 히트맵이 메인. 장식은 방해 금지.
5. **공정성 우선**: 코스메틱/후원은 점수 조작 불허.
6. **네 데이터는 네 것**: 로컬 저장 기본. 동기화는 선택. 표준 파일 포맷.
7. **토지(Land)와 영역(Ward)이 컨테이너다**: 지도 구획은 토지. 내부에 영역. 모든 판은 Primary 토지 1개 소속(없으면 부두).
8. **룬(Rune)으로 가로지른다**: `status/type/priority` 등 구조화된 룬으로 검색·필터·상태 관리.
9. **모아이는 제안하고 사용자는 승인한다**: 석공(요청 시) + 측량사(주간). 자동 변경 기본 금지.
10. **토템(의미)과 장식(꾸미기)을 분리한다**: 토템은 무료 기본 세트로 항상 표현. 유료는 도안(외형)만.
11. **OS로 진화한다**: 노트 → 파일 관리 → IDE(코드를 짠다) → 게임 엔진(세계를 만든다) → 분산 OS. 판게아는 최종적으로 **만들고, 실행하고, 배포하는** 환경이 된다.

## 2) 토지/영역 시스템

### 2.1 토지(Land) = 지도의 땅

- 모든 판은 **Primary 토지 1개** 필수(미지정 시 `부두(Dock)`).
- 기본 렌더는 **활성 Top N(기본: 20)**만 표시, 나머지 `군도(Other)` 압축. N은 설정에서 변경 가능(8/12/20/전부).
- **핀(Pin)**: 활동량 무관 고정 노출. 핀된 토지는 Top N 카운트에 포함되지 않음.
- 온보딩 프리셋: `부두`, `일터`, `배움`, `건강`, `살림`, `재정`, `관계`, `취미` (8개. Top N=20이므로 12개 여유)

### 2.2 영역(Ward) = 토지 내부 구조

- 지도 구역을 만들지 않는다.
- 기본 탐색: 검색 + 최근 + 핀 우선.
- 필요 시 **토지로 승격(Promote)** 가능.

### 2.3 영역 수용량 가드레일

- **직속 판 개수**: 상한 `N=50`, 경고 `80%(=40)` "혼잡" 배지.
- **직속 텍스트 총량**: 경고 `150,000자`, 소프트 `200,000자`. 저장/편집 절대 차단 안 함.
- **상한 초과 시**: 기존 판 편집/저장 차단 금지. 새 판에만 정리 선택지(분류함 자동 생성 / 다른 영역 / 부두).

#### 적용 레벨

| 레벨 | 가드레일 적용 | 설명 |
|------|-------------|------|
| **영역(Ward)** | ✅ 적용 | 해당 영역 직속 판 50개 기준. |
| **토지 루트 (영역 없이 직접 둔 판)** | ✅ 동일 적용 | 토지 루트도 "기본 영역"으로 취급. 직속 판 50개 기준. |
| **토지 전체 합산** | ❌ 미적용 | 토지 내 영역이 10개이고 각 50개면 총 500개 가능. 토지 전체 상한은 없음. |

### 2.4 토지 타입

- **영역(Area)**: 영구(학습/건강/일상).
- **목표(Goal)**: 기한 있음. 완료/기록보관 자연스러움.
- **기록보관(Archive)**: 읽기 중심.

#### 토지 타입 전환 규칙

| 전환 | 허용 | 동작 |
|------|------|------|
| Area → Goal | ✅ | `deadline` 필드 설정 요구. 기존 판/다리 영향 없음. |
| Goal → Area | ✅ | `deadline` 필드 제거. 기존 판/다리 영향 없음. |
| Area → Archive | ✅ | 토지 내 모든 판의 `status`를 `archived`로 일괄 변경 (확인 모달). 대륙 뷰에서 디밍(opacity 0.5). |
| Goal → Archive | ✅ | 동일. "목표 완료" 또는 "보관" 선택지. |
| Archive → Area | ✅ | 판의 `status`는 변경하지 않음 (개별 판에서 수동으로). 대륙 뷰 디밍 해제. |
| Archive → Goal | ✅ | `deadline` 설정 요구 + 디밍 해제. |
| Goal 기한 도래 | 알림만 | 자동 Archive 안 함. "기한이 지났습니다. 완료 처리할까요?" 알림. |

- 전환 경로: 토지 내부 뷰 → ⚙ 설정 → 토지 타입 변경.
- 부두(Dock)는 타입 전환 불가 (항상 Inbox).

### 2.5 룬(Rune) — 가로 분류

- Phase 1 고정 룬:
  - `status`: `inbox | draft | active | blocked | done | archived`
  - `type`: `note | idea | meeting | clip | output | reference`
  - `priority`: `low | medium | high | urgent`
- 룬은 지도 구역을 만들지 않는다. 필터/검색/토템 고도로만 동작.

### 2.6 토지 위생

- 중복 감지/병합 제안(측량사 주간 리포트).
- 영역→토지 승격, 토지→영역 강등.
- 부두는 "분류 대기"(주간 정리에서 비우기 제안).

### 2.7 어명(Decree) — Phase 2+

- 토지의 목적/규칙/시드를 선포하는 정의서. Phase 1에서는 저장만.

### 3.4 온보딩 (FTUE — First Time User Experience)

#### 목표

첫 실행 60초 안에: (1) 첫 판을 쓰고, (2) 바다 위에 섬이 솟아오르는 걸 보고, (3) "이건 다르다" 체감.

#### 플로우

```
[앱 실행] → 빈 바다 + 안개
     │
     ▼
"당신의 대륙을 시작합니다" (풀스크린 타이포)
     │
     ▼
[이름 입력] → "탄고래의 판게아" 생성
     │
     ▼
[프리셋 토지 선택] (다중 선택, 최소 1개)
  ☑ 일터  ☑ 배움  ☐ 건강  ☐ 살림  ☐ 재정  ☐ 관계  ☐ 취미
  + "직접 만들기"
     │
     ▼
[첫 판 작성 유도]
  → 선택한 첫 번째 토지에 빈 판 자동 생성
  → 에디터 열림 + 가이드 오버레이: "아무거나 적어보세요"
  → 3줄 이상 입력 시 "저장" 활성화
     │
     ▼
[대륙 등장 연출]
  → 저장 순간 카메라가 대륙 뷰로 전환
  → 안개가 걷히며 첫 섬이 솟아오름 (0.8초 애니메이션)
  → 섬 위에 방금 쓴 판이 블록 1개로 빛남
  → "첫 번째 땅이 솟아올랐습니다" 토스트
     │
     ▼
[가이드 종료]
  → 모아이(석공) 등장: "더 기록하면 대륙이 자랍니다"
  → 하단 탭 안내 → 자유 탐색
```

#### 원칙

- 계정 생성은 온보딩 **후**에도 가능 (게스트 모드로 시작).
- 프리셋 토지는 나중에 삭제/추가 자유.
- "부두"는 자동 생성 (선택지에 안 보임).
- 온보딩 스킵 가능하지만, 비추 (빈 바다는 차가움).
- 2회차 이후: 바로 대륙 뷰 진입.

## 4) 홈 시각화 — 살아있는 바다 위의 대륙

### 4.0 비전

앱을 열면 **바다 위에 떠 있는 섬들의 세계**가 보인다.
각 섬 = 토지(프로젝트). 섬 위의 작은 블록 하나하나 = 개별 판(노트).
바다는 살아있다 — 파도가 치고, 갈매기가 날고, 구름이 떠다니고, 낮과 밤이 바뀐다.

**기본 뷰는 아이소메트릭(45° 2.5D)**. 블록의 높이감(=데이터 용량)이 산 실루엣으로 한눈에 보인다.

이것은 대시보드가 아니라 **세계**다.

### 4.1 핵심 레퍼런스

| 레퍼런스 | 가져올 것 | 버릴 것 |
|----------|-----------|---------|
| **TradingView 스탁 히트맵** | 면적=크기 비례, 섹터/종목 중첩 구조, 직관적 색상 | 완전 평면, 금융 UI 톤 |
| **Dialogue Theory Block Heatmap** (Dribbble) | 3D 블록 높이감, 셰이더/재질, 도시 스카이라인 | 균일 그리드 (→ 군집 배치로 대체) |
| **David Li Ocean Wave Simulation** (WebGL) | 리얼한 파도/수면 반사, WebGL 쉐이더 기반 바다 | 바다만 있음 (→ 위에 섬 올림) |

### 4.2 데이터 매핑 (확정)

| 시각 요소 | 데이터 | 설명 |
|-----------|--------|------|
| **1 블록 = 1 판** | 개별 노트 | 판(Tablet) 하나가 블록 하나로 렌더. 토지 전체가 아님. |
| **블록 색상(붉기)** | 오늘 활성도 | 초록(조용) → 노랑 → 주황 → 빨강(활발). 오늘 편집/열람 기준. |
| **블록 높이** | 데이터 용량(KB) | 내용이 많은 판일수록 높이 솟음. 높은 판 → 섬 중앙 배치 → **산 실루엣**. |
| **섬 면적** | 판 개수 | 판이 많은 토지가 큰 섬. |
| **측면** | 상단 대비 60~70% 어둡게 | 자동 계산. |

#### 활성도(Activity) 계산 공식

```
activity(tablet, today) = min(1.0, raw_score / cap)

raw_score =
    edit_count × 0.4      (오늘 이 판을 편집한 횟수)
  + view_count × 0.1      (오늘 이 판을 열람한 횟수)
  + char_delta × 0.002    (오늘 추가/삭제된 글자 수의 절대값)

cap = 5.0   (정규화 상한)
```

| 파라미터 | 설명 | 중복 제거 |
|----------|------|----------|
| `edit_count` | 저장(Cmd+S / 자동저장) 1회 = 1카운트 | 같은 세션에서 30초 내 재저장은 1회로 카운트 |
| `view_count` | 에디터 또는 상세 패널에서 판을 연 횟수 | 10분 내 재열람은 1회로 카운트 |
| `char_delta` | `|오늘 글자수 - 어제 글자수|` | - |

- **갱신 타이밍**: 판 저장 시 실시간 업데이트. 자정(00:00) 리셋.
- **저장**: 로컬 `activity_log.json` — `{tablet_id, date, edit_count, view_count, char_delta}`.
- **어제 이전 데이터**: 3일 보관 후 삭제 (활성도는 "오늘"만 사용. Landmass 계산은 별도).
- **활성도 0**: 오늘 아무 행위도 없는 판. 가장 차가운 색(초록/청록).

#### 색상 보간표 (활성도 → 블록 상단색)

| 활성도 | 낮 모드 RGB | 밤 모드 RGB | 설명 |
|--------|-----------|-----------|------|
| 0.00 | `rgb(95, 158, 105)` | `rgb(28, 52, 72)` | 조용 (초록/청록) |
| 0.25 | `rgb(148, 190, 72)` | `rgb(42, 82, 92)` | 약간 활성 (연두) |
| 0.50 | `rgb(220, 205, 58)` | `rgb(85, 125, 62)` | 보통 (노랑) |
| 0.75 | `rgb(245, 155, 48)` | `rgb(205, 135, 48)` | 활발 (주황) |
| 1.00 | `rgb(225, 68, 42)` | `rgb(235, 88, 52)` | 매우 활발 (빨강) |

- 중간값: 선형 보간(linear interpolation).
- **측면색**: 상단색 × 0.55 (RGB 각 채널).
- **색맹 대응**: 색상 외에 블록 높이(용량)가 독립 채널이므로, 색상 단독 의존 아님. Phase 2+에서 대체 팔레트(청황) 제공.

### 4.3 블록·그리드 수치

#### 블록 크기 (아이소메트릭 기준)

| 항목 | 값 | 설명 |
|------|-----|------|
| 블록 기본 크기 | 18×18px (탑다운 기준) | 아이소메트릭 변환 후 약 18×9px 다이아몬드 |
| 블록 간 간격 | 1px | 밀착. 섬 내부 판 사이. |
| 블록 높이 최소 | 2px | activity=0, volume=0인 판 |
| 블록 높이 최대 | 11px | volume 기준: `height = 2 + min(9, volume_kb / 20 × 9)` |
| 블록 측면 색상 | 상단 × 0.55 (어둡게) | 자동 계산 |

#### 아이소메트릭 변환

```
// 탑다운 좌표 (x, y) → 아이소메트릭 화면 좌표 (sx, sy)
sx = (x - y) × cos(30°) × scale    // ≈ (x - y) × 0.866 × scale
sy = (x + y) × sin(30°) × scale    // ≈ (x + y) × 0.5 × scale

// z-order (깊이 정렬): y값이 큰 블록이 앞에 (화가 알고리즘)
z_index = y × grid_cols + x
```

#### 섬 간 간격

| 항목 | 값 | 설명 |
|------|-----|------|
| 섬 간 최소 간격 | 블록 크기 × 3 = 54px | 바다가 충분히 보여야 함 |
| 섬 테두리 패딩 | 4px | 테두리선과 최외곽 블록 사이 |
| 섬 라벨 오프셋 | 섬 상단 -20px | 라벨 위치 |

#### 군도(Other) 압축

- 롱테일 토지 (활성 Top N 이후) → 군도로 합침.
- **α=0.6은 블록 투명도**: 군도 블록은 `opacity: 0.6` 렌더.
- 군도 블록 크기는 동일 (18×18px). 간격도 동일.
- 군도 라벨: "기타 (3개 토지, 14판)".

### 4.4 섬 배치(레이아웃) 알고리즘

#### Phase 1: 가중 그리드 배치

```
입력: lands[] — 각 토지의 {id, tablet_count, total_volume, pinned}
출력: lands[].position — 각 토지의 (x, y) 좌표

1. 정렬
   - pinned 토지를 먼저, 나머지는 tablet_count 내림차순.
   - 부두(Dock)는 항상 마지막.

2. 섬 크기 계산
   - cols = ceil(sqrt(tablet_count × 1.15))
   - rows = ceil(tablet_count / cols)
   - island_width  = cols × (block_size + gap)
   - island_height = rows × (block_size + gap)

3. 배치 (행 기반 래핑)
   - canvas_width = viewport_width / zoom (논리 폭)
   - cursor = (margin_left, margin_top)
   - for each land:
       if cursor.x + island_width > canvas_width:
           cursor.x = margin_left
           cursor.y += max_row_height + island_gap
       land.position = cursor
       cursor.x += island_width + island_gap
       max_row_height = max(max_row_height, island_height)

4. 줌 핏 계산
   - bounding_box = 모든 섬을 감싸는 최소 사각형
   - zoom_fit = min(viewport_w / bb_w, viewport_h / bb_h, 2.5)
   - pan = 중앙 정렬
```

#### 수동 배치 모드

- 자동 배치 대신 `land_layout.json`의 좌표를 사용.
- 새 토지 추가 시: 가장 큰 빈 영역의 중앙에 배치.
- 충돌(겹침) 감지: 섬 바운딩 박스 교차 시 스냅으로 밀어냄.

- 같은 토지의 판들이 밀착(1px 간격)으로 군집 → 자연스럽게 **섬** 형태.
- 섬 간에는 넉넉한 간격(54px+) → 바다가 보임.
- 섬 주변 **테두리선**(반투명) → 토지 경계 명확.
- 롱테일 토지는 **군도(Other)** — 블록 opacity 0.6.

### 4.6 산 형태 배치 (기본 정렬)

- 기본 정렬 = **"산(용량순)"**: 용량 큰 판이 섬 **가운데**, 작은 판이 **바깥**으로.
- 자연스럽게 산봉우리 실루엣이 만들어짐.
- Spiral-center 알고리즘: 정렬 → 가장 큰 값을 중앙 인덱스 배치 → 나머지 좌우 교대.

### 4.7 정렬 모드 (사용자 전환 가능)

| 정렬 | 키 | 중앙 배치 기준 | 효과 |
|------|----|---------------|------|
| **산(용량)** | `mountain` | 용량(KB) 큰 판 → 가운데 | 산 실루엣 (기본) |
| **활성도** | `activity` | 활발한 판 → 가운데 | 뜨거운 중심 |
| **최신** | `recent` | 새로 만든 판 → 가운데 | 새 기록 중심 |
| **상태별** | `status` | status 룬 그룹핑 | 상태별 구역 |
| **유형별** | `type` | type 룬 그룹핑 | 유형별 구역 |

### 4.8 바다(Ocean) — WebGL 리얼 시뮬레이션

**목표**: David Li 수준의 리얼한 바다. SVG 가짜 파도가 아니라 진짜 바다.

**구현 방향:**
- **WebGL/Three.js 기반 수면 쉐이더**
  - 물결: Gerstner Wave 또는 FFT 기반 파도 시뮬레이션.
  - 수면 반사: 환경 큐브맵 + 프레넬 반사.
  - 수면 굴절: 아래 깊은 바다색이 비침.
  - 폼(Foam): 파도 꼭대기에 흰 거품.
- **낮/밤 전환** → **시간대별 조명 시스템 (4.8.1 참조)**
- **대기 효과**
  - 구름: 3D 빌보드 또는 파티클. 천천히 이동. 시간대에 따라 색 변화.
  - 갈매기: 스프라이트 애니메이션. 새벽~일몰에만.
  - 안개(원거리): 먼 섬일수록 안개로 흐려짐 (깊이감). 새벽에 가장 짙음.
  - 별: 일몰 후 서서히 등장. 자정 최대 밝기. 새벽 전 페이드 아웃.

| 7 | **일몰** | 17:00~19:00 | 서쪽 수평선 | `#D4784A` → `#2A1530` | `#1A5A80` → `#0D1B30` | 0.4 |
| 8 | **밤** | 19:00~00:00 | 없음 (달) | `#0A1628` | `#0A1628` | 0.15 |

> 시간대 간 전환은 **선형 보간**. 갑자기 변하지 않고 서서히 흘러간다.
> 조도(illuminance) 0.0~1.0: 모든 시각 요소의 밝기 베이스라인.

##### 태양·달 위치와 그림자

```
태양 위치 (방위각 + 고도):
  azimuth  = (hour - 6) / 12 × 180°     // 동(0°) → 남(90°) → 서(180°)
  altitude = sin((hour - 6) / 12 × π) × max_altitude

  max_altitude = 65°  (하지 기준. 계절 무시 — 판게아는 항상 여름.)

달 위치:
  azimuth  = (hour - 18) / 12 × 180°    // 일몰 후 동 → 자정 남 → 새벽 서
  altitude = sin((hour - 18) / 12 × π) × 45°
```

##### 블록 그림자

| 항목 | 값 | 설명 |
|------|-----|------|
| **그림자 방향** | 태양/달 azimuth의 반대 방향 | 태양이 동쪽이면 그림자는 서쪽. |
| **그림자 길이** | `block_height × (1 / tan(altitude))` × 0.4 | 태양이 낮을수록 그림자 길다. 0.4 = 과장 방지 계수. |
| **그림자 색상** | `rgba(0, 0, 0, 0.08 × (1 - illuminance × 0.5))` | 정오에 가장 옅고, 일출/일몰에 가장 진함. |
| **그림자 blur** | `shadow_length × 0.3` | 긴 그림자일수록 부드럽게. |
| **달빛 그림자** | 태양 그림자 × 0.3 (약하게) | 밤에도 미세한 그림자. |

##### 섬 그림자 (바다 위 투영)

- 섬 전체가 바다 위에 떠 있으므로, **섬 아래 바다에 큰 그림자**가 드리운다.
- 섬 그림자: 섬 바운딩 박스를 태양 방향으로 오프셋 + blur.
- 정오: 섬 바로 아래 (짧고 진한 그림자).
- 일출/일몰: 섬에서 멀리 늘어진 긴 그림자 (옅고 긴).
- 밤: 달빛 그림자 (매우 옅음) 또는 없음.

```
island_shadow_offset_x = cos(sun_azimuth) × island_width × 0.3 × (1 - altitude/90°)
island_shadow_offset_y = sin(sun_azimuth) × island_height × 0.15 × (1 - altitude/90°)
island_shadow_blur = island_width × 0.5 × (1 - altitude/90°)
island_shadow_opacity = illuminance × 0.15
```

##### 수면 반사

| 시간대 | 반사 효과 |
|--------|----------|
| **일출/일몰** | 태양 방향에서 강한 오렌지/골드 반사 줄기 (sun path). 가장 드라마틱. |
| **정오** | 넓고 밝은 반사. 스파클(반짝이는 점) 최대. |
| **밤** | 달빛 반사 줄기 (moon path). 은색. 좁고 길게. |
| **새벽** | 수면 전체가 미세하게 분홍/보라 틴트. 반사 약함. |

```
// 반사 줄기 (sun/moon path on water)
reflection_x = viewport_center_x + cos(azimuth) × distance_to_horizon
reflection_y = water_surface_y
reflection_width = 120px × (1 - altitude/90°)  // 낮은 각도일수록 넓음
reflection_intensity = illuminance × 0.15        // 태양: 0.15, 달: 0.05
```

##### 성능 고려

| 품질 | 그림자 | 반사 | 대기 |
|------|--------|------|------|
| **최대** | 블록별 개별 그림자 + 섬 그림자 | 수면 반사 줄기 + 스파클 | 전부 |
| **보통** | 섬 단위 통합 그림자만 | 단순 반사 글로우 | 구름/갈매기만 |
| **최소** | CSS box-shadow (고정 방향) | 그라데이션 오버레이 | 별만 (CSS) |

> 최소 품질에서도 "시간이 흐른다"는 느낌은 유지: 배경색 그라데이션 + 별 페이드 + 구름 색 변화.

- **성능 타협 — 3단계 품질 폴백**:

  | 품질 | 렌더 방식 | 대상 | FPS 기준 |
  |------|----------|------|---------|
  | **최대** | WebGL Gerstner Wave + 반사 + 폼 + 파티클 | 데스크탑 GPU | 60fps |
  | **보통** | WebGL 단순 파도 (반사 생략, 파티클 축소) | 모바일 고사양 | 45fps |
  | **최소** | Canvas 2D 파도 + CSS 그라데이션 | 모바일 저사양 | 30fps |

  - **자동 품질 전환**: 첫 3초간 FPS 측정 → 평균 45fps 미만 시 한 단계 하향. 사용자 수동 설정도 가능.
  - **최소 품질도 "바다"여야 한다**: CSS 파도여도 물결 애니메이션 + 색상 그라데이션 + 낮/밤 전환은 유지. "정적 배경"이 아니라 "움직이는 바다".
  - M1(코어)에서는 **Canvas 2D 파도로 출시** → M3에서 WebGL 업그레이드. 기본 경험부터 확보.

### 4.11 수동 배치 & 꾸미기 모드

#### 배치 모드

대륙 뷰에서 섬(토지)과 판(블록)의 위치를 사용자가 자유롭게 조정할 수 있다.

| 대상 | 배치 방법 | 설명 |
|------|----------|------|
| **섬(토지) 위치** | 편집 모드 → 섬 드래그 | 섬 전체를 바다 위 원하는 곳으로 이동. 다른 섬과 겹침 방지(스냅). |
| **판(블록) 위치** | 편집 모드 → 블록 드래그 | 섬 내에서 블록 순서/위치를 수동 조정. 산 정렬 해제 후 자유 배치. |
| **섬 간 간격** | 편집 모드 → 섬 테두리 드래그 | 섬 사이 바다 간격 조정. |

#### 편집 모드 진입

- 상단 바 **🖊 편집** 버튼 또는 `Cmd+E`.
- 편집 모드 활성화 시: 모든 섬/블록에 이동 핸들 표시 + 배경 격자 가이드 (스냅 보조).
- 편집 모드 종료: **✓ 완료** 또는 `Escape` → 위치 저장.

#### 배치 옵션

| 설정 | 설명 | 기본값 |
|------|------|--------|
| **자동 정렬** | 시스템이 판 개수/용량에 따라 배치 (산/활성도/최신 등). | ON |
| **수동 배치** | 사용자가 직접 배치한 위치를 유지. 새 판 추가 시 빈자리에 배치. | OFF |
| **혼합** | 섬 위치는 수동, 섬 내부 블록은 자동 정렬. | - |

> 수동 배치 선택 시, 정렬 드롭다운(산/활성도 등)은 비활성화.
> "자동 정렬로 되돌리기" 버튼으로 언제든 시스템 배치로 복원 가능.

#### 저장

- 섬 위치: `land_layout.json` — `{land_id, x, y}` 배열.
- 블록 위치: 각 토지별 `block_layout.json` — `{tablet_id, row, col}` 배열.
- 동기화 시 위치 데이터도 함께 Sync.

## 5) 토템 & 장식(지도 상징물)

### 5.1 설계 원칙(잠금)

1. 토템(의미)과 장식(꾸미기) 분리.
2. 의미는 무료 기본 세트로 표현 가능.
3. 유료는 도안(외형)만. 점수/가시성 영향 0%.
4. 토템은 고도(Altitude) 1개만 동시 표시(기본: `status`=봉화).
5. **토템은 개별 판(블록) 위에 표시**된다. 줌 레벨에 따라 가시성 조절.
6. 토템 계산은 클라이언트 로컬에서만.
7. 토템/장식 렌더 오류 → 판 저장/편집 영향 없음.
8. 모노리스는 토지(섬) 단위로 섬 중심에 1개 존재.
9. **사용자가 토템·장식·모노리스를 개별 온/오프** 할 수 있다.

### 5.1.1 표시 토글 (사용자 설정)

### 5.4 토템 표시 로직

#### 개별 판 위의 토템 (v29.3 확정)

- 각 판 블록 위에 **해당 판의 룬 값에 맞는 토템 아이콘**이 표시된다.
- 예: 봉화(status) 고도일 때, `status:active` 판 위에 🔥, `status:done` 판 위에 🏛.
- 줌 레벨/호버/활성도에 따라 가시성 조절 (4.8 LOD 참조).

#### 모노리스 위의 대표 토템 (섬 단위 요약)

모노리스는 토지(섬)를 대표하는 석조물. 모노리스 위에 토지 전체를 요약하는 **대표 토템**을 배치한다.

`share(value) = count(value) / total_tablets`

- **Primary**: `share(top1) >= 0.40` → 대표 토템. 미달 시 혼합 아이콘.
- **최소 임계치**: 판 < 5개 → 대표 토템 생략, 모노리스만.
- **Secondary**(최대 2개): `share >= 0.25`. Primary와 동일 키면 생략.

> 정리: 개별 판 = 자기 룬 토템. 모노리스 = 토지 전체 요약 토템. 두 체계가 공존.

### 5.5 배치(결정적)

- 좌표: `[0..1]²`. 모노리스: `hash(land_id)` 고정. 토템: 6슬롯 해시.
- 겹침: `d_min` → 다음 슬롯(3회) → 2D 폴백.

### 5.6 장식(Ornament) — Phase 2+

- 의미 0. 기본 무료 + 도안 유료. zoom-out 20 / zoom-in 80.

### 5.7 도안(Codex)

- semantic_key 고정. 외형만 대체. 미보유 시 기본 폴백. 접근성: 형태+기호.

## 6) 대륙 성장(성취 루프)

### 6.1 Landmass — Phase 1 기본값

| 신호 | 가중치 | 설명 |
|------|--------|------|
| 기여(새 판/정리) | 1.0 | 새 판 + 구조 정리 |
| 유지(재방문/수정) | 0.8 | 열람·수정(당일 중복 제외) |
| 연결(다리/교차) | 1.2 | 토지 간 다리·흔적 생성 |
| 정리(기록보관/분류) | 0.6 | 부두→토지, 기록보관 |
| 스팸/중복 | -0.5 | 반복 감쇠 |

> feature flag A/B. Landmass는 순수 활동 지표. 코스메틱/후원 완전 분리.

#### 계산 로직

```
landmass(land) = Σ events_today × weight

이벤트 기록:
- 새 판 생성            → {type: "create",  land_id, tablet_id, ts}
- 판 편집(저장)         → {type: "edit",    land_id, tablet_id, ts}
- 판 열람              → {type: "view",    land_id, tablet_id, ts}
- 다리 생성            → {type: "bridge",  land_id, target_land_id, ts}
- 부두→토지 배정        → {type: "assign",  land_id, tablet_id, ts}
- 기록보관             → {type: "archive", land_id, tablet_id, ts}
```

#### 스팸 감쇠 규칙

| 조건 | 감쇠 |
|------|------|
| 같은 판을 5분 내 재편집 | 2회차부터 가중치 0 (카운트 안 됨) |
| 같은 판을 10분 내 재열람 | 2회차부터 가중치 0 |
| 판 생성 후 60초 내 삭제 | 기여 점수 회수 (−1.0) |
| 하루 판 생성 20개 초과 | 21번째부터 가중치 0.2로 감쇠 |

#### 저장·갱신

| 항목 | 설명 |
|------|------|
| **갱신 타이밍** | 이벤트 발생 시 즉시 (로컬). 서버 Sync는 5분 배치. |
| **저장 위치** | 로컬: `landmass_log.json` — 일별 이벤트 로그. 서버: 일별 집계값만. |
| **보관 기간** | 로컬: 30일 롤링. 서버: 영구 (일별 집계). |
| **리셋** | 일별 집계. 자정(00:00 로컬 타임존) 기준 새 날 시작. |
| **누적 Landmass** | `total_landmass(land) = Σ daily_landmass` (전체 기간 누적). |

#### Landmass와 시각화의 관계

| 시각 요소 | Landmass 영향 | 실제 데이터 소스 |
|-----------|-------------|-----------------|
| 블록 색상 | ❌ 없음 | 오늘 활성도(activity) |
| 블록 높이 | ❌ 없음 | 판 데이터 용량(KB) |
| 섬 면적 | ❌ 없음 | 판 개수(tablet_count) |
| 섬 배치 순서 | ❌ 없음 | tablet_count 내림차순 |
| **성취 뱃지 / 퀘스트** | ✅ 사용 | 총 Landmass 기준 업적 해금 |
| **프로필 레벨 (Phase 2+)** | ✅ 사용 | 대륙 전체 Landmass 합산 → 레벨 |
| **은하계 랭킹 (Phase 2+)** | ✅ 사용 | 공개 Landmass 비교 |

> **핵심**: Landmass는 **순수 성취 지표**. 블록 렌더링에 직접 영향을 주지 않는다. 시각화는 실시간 데이터(활성도/용량/판수)로만 결정된다.

## 7) 교차 정착 & 흔적(Trace)

- 판은 Primary 토지 1곳에만 원본. 다른 토지 참조 → 흔적.
- 점선 다리로 원본↔흔적 연결.
- 기본 3개 흔적만 표시. 확대 시 추가.

#### 흔적 시각화

| 속성 | 원본 블록 | 흔적 블록 |
|------|----------|----------|
| **투명도(opacity)** | 1.0 | 0.45 |
| **크기** | 18×18px (기본) | 동일 (18×18px) |
| **높이** | 용량 기반 정상 높이 | 원본과 동일 높이 |
| **테두리** | 없음 (또는 선택 시 금색) | 점선 테두리 (1px, 투명도 0.5) |
| **색상** | 활성도 기반 정상 색상 | 동일 색상 + 투명도 적용 |
| **토템** | 정상 표시 | 표시하지 않음 |
| **호버/탭** | 상세 패널 표시 | 상세 패널 + "원본 보기" 링크 |

#### 흔적 표시 규칙

| 규칙 | 설명 |
|------|------|
| **기본 표시 수** | 토지당 최대 3개 흔적 표시 (activity 높은 순). |
| **줌 확대 시** | 줌 ≥ 1.6x → 최대 10개. 줌 ≥ 3x → 전부 표시. |
| **원본↔흔적 다리** | 점선 SVG path (strength 무관, 고정 스타일). |
| **흔적 클릭** | 상세 패널 + "원본으로 이동" 버튼 (원본 토지로 줌 인). |
| **흔적 삭제** | `xset[]`에서 해당 항목 제거. 다리도 함께 삭제. |

### 7.1 다리(Bridge) 로직

#### 다리 생성 조건

| 트리거 | 설명 | 방향 |
|--------|------|------|
| `[[판 제목]]` 또는 `[[판 ID]]` 링크 | 판 A에서 판 B를 참조하면, 토지 A→토지 B에 다리 생성 | 단방향 |
| `xset[]` 교차 정착 | 판이 다른 토지에 흔적을 남기면, 원본 토지↔흔적 토지에 다리 생성 | 양방향 |
| 측량사 다리 추천 승인 | 모아이가 추천한 다리를 사용자가 승인 | 양방향 |

#### 다리 규칙

```
bridge(land_A, land_B) = {
    source_land_id,
    target_land_id,
    link_count,       // A→B 방향 링크 수
    strength,         // min(1.0, link_count / 5) — 시각적 굵기에 사용
    created_at,
    updated_at
}
```

| 규칙 | 설명 |
|------|------|
| **같은 토지 내 링크** | 다리 아님. 영역 간 참조일 뿐. 다리는 **토지 간**만. |
| **방향** | 링크 기반 = 단방향. 교차 정착 = 양방향. 양쪽 모두 링크 → 양방향 표시. |
| **합산** | 같은 토지 쌍의 링크는 하나의 다리로 합산. `link_count` 증가. |
| **삭제** | 링크 삭제 → `link_count - 1`. 0이 되면 다리 삭제. |
| **교차 정착 해제** | `xset[]`에서 제거 → 다리 삭제 (교차 정착 기반 다리만). |
| **토지 단위 최대** | 한 토지에서 나가는 다리 최대 20개. 초과 시 strength 낮은 것부터 비표시. |

#### 다리 시각화

| strength | 선 스타일 | 설명 |
|----------|----------|------|
| 0.0~0.2 | 점선, 1px, 투명도 0.2 | 약한 연결 |
| 0.2~0.5 | 점선, 1.5px, 투명도 0.4 | 보통 연결 |
| 0.5~1.0 | 실선, 2px, 투명도 0.6 | 강한 연결 |

- SVG `<path>`: 이차 베지어 곡선. 제어점 = `(cx, min(y1,y2) - 28 - strength × 18)`.
- 항로(bridges) 고도에서만 기본 표시. 다른 고도에서는 선택된 섬의 다리만.

## 8.5) 판 에디터

### 8.5.1 설계 원칙

1. **속도 최우선**: 열기 ≤ 200ms. 타이핑 지연 0. 오프라인 완전 동작.
2. **마크다운 네이티브**: 내부 포맷 = 마크다운. WYSIWYG가 아니라 실시간 렌더.
3. **모바일에서도 쓸 만하다**: 터치 최적화 툴바. 스와이프 뒤로가기 = 저장+닫기.
4. **부가 기능은 모아이가 한다**: 에디터 자체는 가볍게. AI 기능은 석공 호출.

### 8.5.2 에디터 모드

| 모드 | 설명 | 기본 |
|------|------|------|
| **라이브 프리뷰** | 입력하면 즉시 렌더. 헤딩·볼드·리스트가 실시간 변환. | ON (기본) |
| **소스 모드** | 순수 마크다운 텍스트. 코드펜스·front-matter 직접 편집. | 토글 |
| **읽기 모드** | 렌더만. 편집 불가. 공유·프레젠테이션용. | 토글 |

### 8.5.3 에디터 기능

| 기능 | Phase | 설명 |
|------|-------|------|
| **마크다운 문법** | 1 | 헤딩/볼드/이탤릭/리스트/체크박스/인용/코드/테이블/수평선 |
| **실시간 렌더** | 1 | 타이핑 즉시 반영. 코드블록 하이라이팅. |

| **태그** | 1 | `#태그` → 룬과 별도의 자유 태그. 검색에서 활용. |

| **동시 편집 잠금** | 다른 기기에서 같은 판 편집 중이면: "다른 기기에서 편집 중입니다" 표시. 강제 편집 가능(충돌 판 생성). |

| 채널 | Phase | 설명 |

| 이벤트 | 우선순위 | 설명 |
|--------|----------|------|
| **측량사 주간 리포트** | 보통 | 매주 월요일(설정 가능). 인앱 + 선택적 푸시. |
| **부두 적체 경고** | 보통 | 부두 판 ≥10개. "정리할 시간" 넛지. |

### 8.7.3 원칙

- **기본 최소**: Phase 1에서는 인앱 뱃지 + 선택적 푸시만. 과도한 알림 금지.
- **조용한 기본값**: 모든 푸시는 OFF가 기본. 사용자가 명시적으로 ON.
- **리마인더는 룬**: `remind_at` 필드가 front-matter에 추가됨. 날짜 지나면 자동 소멸 (알림만 보내고 판 변경 없음).
- **알림 없이도 완전 동작**: 알림은 편의 기능. 핵심 루프에 영향 없음.

- 공유된 판/토지는 **토템 고도 기본 OFF** (개인 분류 구조 비공개).

- AI(모아이) 기능은 공유 뷰에서 동작하지 않음 (내 잉크 보호).
- front-matter의 민감 룬(priority 등)은 공유 시 선택적 숨김.

### 8.8.4 공유 링크 구현

#### URL 구조

```
읽기전용 링크:  https://pangaea.app/s/{share_id}
임베드:        https://pangaea.app/embed/{share_id}

share_id = base62(random 12bytes)  // 예: "a3Kf9mWx2qPL"
```

#### 공유 데이터 모델

```json
{
  "share_id": "a3Kf9mWx2qPL",
  "owner_id": "user_01J...",
  "tablet_id": "01J5K3...",
  "type": "tablet",
  "permissions": "read",
  "hide_runes": ["priority"],
  "hide_totem": true,
  "expires_at": "2025-07-01T00:00:00Z",
  "created_at": "2025-06-01T12:00:00Z",
  "revoked": false
}
```

#### 만료·무효화 로직

| 시나리오 | 동작 |
|----------|------|
| **만료 기한 도래** | 서버에서 `expires_at` 체크. 만료 시 404 + "이 링크는 만료되었습니다" 페이지. |
| **수동 비공개 전환** | `revoked = true` → 즉시 무효화. 기존 URL 접속 시 404. |
| **판 삭제** | 원본 판 삭제 시 공유 링크도 자동 무효화. |
| **만료 기한 옵션** | 무기한(기본) / 1일 / 7일 / 30일 / 커스텀 날짜. |
| **공유 관리** | 설정 → 공유 관리 → 활성 공유 링크 목록 + 만료일 변경 + 비공개 전환. |

#### 공유 페이지 렌더링

- 서버사이드 렌더링 (SSR). 마크다운을 HTML로 변환하여 서빙.
- OG 메타 태그 자동 생성 (제목, 첫 200자 설명, 대표 이미지).
- 공유 페이지 하단: "PANGAEA에서 작성됨" + CTA 링크.

## 9) 파일 시스템 & 확장자

### 9.1 기본 원칙

- **판(Tablet) = 파일**. 파일이 기본 단위다.
- 기본 생성 형식은 `.md` (Markdown + front-matter).
- 그 외 모든 확장자도 판으로 등록 가능.
- front-matter는 `.md` 판에 내장. 비-md 판은 사이드카 파일(`.meta.json`)로 메타데이터 관리.

### 9.2 확장자 지원 단계

| Phase | 지원 확장자 | 기능 | 설명 |
|-------|------------|------|------|
| **Phase 1** | `.md` | 작성·편집·렌더·검색 | 마크다운 중심. 즉시 렌더. 로컬 퍼스트. |
| **Phase 1** | `.png/.jpg/.gif/.svg` | 첨부·미리보기 | 이미지 판. 대륙에서 시각적 블록으로 표시. |
| **Phase 1** | `.pdf` | 첨부·미리보기 | 문서 판. 임포트 시 자동 등록. |
| **Phase 2** | `.csv/.xlsx` | 테이블 뷰·기본 편집 | 데이터 판. 간단한 스프레드시트 뷰. |
| **Phase 2** | `.json/.yaml/.toml` | 구조화 뷰·편집 | 설정/데이터 판. 트리 뷰 제공. |
| **Phase 2** | `.html` | 렌더·미리보기 | 웹 판. 인라인 미리보기. |
| **Phase 2** | `.py/.js/.ts` 등 | 코드 하이라이팅·편집 | 코드 판. 모나코 에디터 연동. |
| **Phase 3** | `.canvas` | 비주얼 캔버스 | Obsidian Canvas 호환. 판 간 연결 시각화. |
| **Phase 3** | `.app/.wasm` | **실행** | 앱 판. 판게아 내에서 미니앱 실행. |
| **Phase 3** | 프로젝트 토지 (IDE) | **빌드·실행·디버그** | 토지 = 프로젝트. 내장 터미널 + LSP + Git. |
| **Phase 3+** | 플러그인 확장자 | 커뮤니티 정의 | 플러그인이 새 확장자 렌더러를 등록. |
| **Phase 4** | `.scene/.prefab` | **게임 엔진** | 씬 에디터 + 물리/렌더 파이프라인. 토지 = 월드. |
| **Phase 4** | `.shader/.glsl` | 쉐이더 편집·미리보기 | 실시간 쉐이더 그래프 + 미리보기. |

### 9.3 판의 분류 체계

| 판 종류 | 확장자 예시 | 대륙 블록 표현 | 아이콘 |
|---------|-----------|---------------|--------|
| 기록 판 | `.md` | 기본 블록 | 📝 |
| 이미지 판 | `.png/.jpg/.svg` | 컬러 썸네일 블록 | 🖼 |
| 문서 판 | `.pdf` | 문서 아이콘 블록 | 📄 |
| 데이터 판 | `.csv/.xlsx/.json` | 격자 무늬 블록 | 📊 |
| 코드 판 | `.py/.js/.ts` 등 | 터미널 색상 블록 | 💻 |
| 앱 판 | `.app/.wasm` | 실행 가능 블록 (발광) | ⚡ |
| 프로젝트 판 | IDE 토지 전체 | 복합 구조물 (성채) | 🏗 |
| 씬 판 | `.scene/.prefab` | 3D 미니어처 블록 | 🎮 |

> 대륙 시각화에서 확장자별로 블록의 **질감/아이콘**이 달라져서, 한눈에 어떤 종류의 파일이 많은지 보인다.

### 9.5 OS 진화 로드맵

```
Phase 1 ─── 마크다운 노트앱: .md 작성/편집, 이미지·PDF 첨부, 대륙 시각화, 토템, 모아이 — "기록하는 도구"
Phase 2 ─── 파일 관리 플랫폼: 코드/데이터/웹 판 편집, GitHub 연동, 플러그인 API 시작 — "판게아 = 내 모든 파일의 집"
Phase 3 ─── 개인 지식 OS + IDE: 앱 판(.app/.wasm) 실행, 내장 IDE(터미널+LSP+Git+디버거), 토지=프로젝트, 빌드·실행·배포 완결, 플러그인 마켓 — "판게아 위에서 코드를 짜고 앱을 돌린다"
Phase 4 ─── 크리에이터 OS + 게임 엔진: 씬 에디터+물리+렌더러, .scene/.prefab/.shader 판, 토지=월드, 3D/2D 에셋 파이프라인, 모아이 에셋/코드 생성 보조 — "판게아 위에서 게임을 만들고 플레이한다"
Phase 5+ ── 분산 OS: 온체인 인덱스, P2P 동기화, 판게아 연합, 분산 배포 — "대륙이 서로 연결된 우주가 된다"
```

> **핵심**: 각 Phase에서 판의 "할 수 있는 것"이 확장된다.

### 9.6 IDE 환경 (Phase 3)

판게아의 IDE는 **토지를 프로젝트로 여는 것**이다. 별도 IDE 앱이 아니라, 토지 뷰가 IDE 모드로 전환된다.

| 기능 | 설명 | 레퍼런스 |
|------|------|----------|
| **에디터** | 모나코 기반 코드 편집. 멀티 탭. 분할 뷰. | VS Code |
| **터미널** | 내장 터미널. 토지 루트가 작업 디렉토리. | xterm.js |
| **LSP** | 언어 서버 프로토콜. 자동완성·진단·정의 이동. | Phase 3 초기: JS/TS/Python |
| **Git 통합** | 커밋·푸시·풀·브랜치·diff. GitHub 연동과 자연스럽게 연결. | Phase 2 GitHub 확장 |
| **디버거** | 브레이크포인트·스텝·변수 감시. WASM 디버깅 포함. | DAP 기반 |
| **빌드 시스템** | 어명(Decree)에 빌드 스크립트 정의. 토지 내에서 빌드·실행. | package.json / Makefile 호환 |
| **모아이 코딩** | 석공이 코드 생성·리팩터링·테스트 생성. Copilot 수준. | AI pair programming |

> 대륙 뷰에서 "프로젝트 토지"는 **성채(Citadel)** 형태로 표현 — 일반 섬보다 복잡한 구조물.

### 9.7 게임 엔진 (Phase 4)

판게아의 게임 엔진은 **토지를 월드로 여는 것**이다. 토지 안에서 게임을 설계하고, 판게아 안에서 플레이한다.

| 기능 | 설명 | 레퍼런스 |
|------|------|----------|
| **씬 에디터** | 노드 기반 씬 트리. 드래그&드롭 배치. 2D/3D. | Godot / Unity |
| **물리 엔진** | Rapier(WASM) 기반. 충돌·중력·조인트. | Rapier.rs |
| **렌더 파이프라인** | WebGPU/WebGL. PBR 머티리얼. 라이팅. | Three.js / Bevy |
| **쉐이더 그래프** | 노드 기반 비주얼 쉐이더 편집. 실시간 미리보기. | Shader Graph |
| **에셋 파이프라인** | `.glb/.gltf/.png/.ogg` → 최적화된 런타임 포맷. | 자동 압축·아틀라스 |
| **스크립팅** | JS/TS/Lua 스크립트 + 비주얼 스크립팅(노드 그래프). | GDScript 대안 |
| **플레이 모드** | 에디터에서 즉시 플레이. 핫 리로드. | Play-in-Editor |
| **모아이 에셋** | 석공이 3D 모델·텍스처·사운드 생성. 프롬프트 → 에셋. | AI asset generation |
| **배포** | 토지를 `.app` 판으로 export → 다른 판게아에서 플레이. | Phase 5 분산 배포 |

> 대륙 뷰에서 "게임 토지"는 **살아있는 미니어처** — 실제 게임 씬의 축소판이 섬 위에 렌더.
> 게임 토지를 클릭하면 플레이 모드로 진입.

## 10) 모아이 & AI

### 10.0 모아이란 무엇인가

**모아이(Moai)** 는 판게아의 **통합 AI 에이전트**다. 섬의 수호신.

사용자는 모아이 **하나**와 대화한다. 모아이가 내부적으로 적절한 **스킬(기능체)**을 선택해서 실행한다. 사용자는 스킬을 의식할 필요가 없다 — "모아이야, 이거 해줘"라고 말하면 된다.

```
사용자: "이 판 요약해줘"
   └→ 모아이 (에이전트) └→ [석공:요약] 스킬 선택 → 실행 → 결과 반환
사용자: "이 토지에 로그인 기능 추가해줘"
   └→ 모아이 (에이전트) └→ [코드생성] 스킬 선택 → 다중 파일 생성 → diff 미리보기
사용자: "중세 마을 3D 모델 만들어줘"
   └→ 모아이 (에이전트) └→ [에셋생성:3D] 스킬 선택 → .glb 생성 → 에셋 라이브러리에 추가
```

### 10.1 아키텍처: 에이전트 + 스킬

| 계층 | 역할 | 비유 |
|------|------|------|
| **모아이 (에이전트)** | 사용자와 대화하는 유일한 인터페이스. 의도를 파악. | Claude 자체 |
| **스킬 (기능체)** | 특정 작업을 수행하는 모듈. 교체/추가/삭제 가능. | Claude의 Tool / MCP Server |

### 10.3 커스텀 스킬 (Phase 3+)

Phase 3부터 사용자/커뮤니티가 **커스텀 스킬**을 만들 수 있다.

```json
{
  "skill_id": "custom:weekly-standup",
  "name": "주간 스탠드업 생성기",
  "description": "이번 주 활동을 분석해 스탠드업 보고서를 만듭니다",
  "author": "tangolae",
  "trigger": "request",
  "input": { "context": "current_land", "time_range": "7d" },
  "prompt_template": "이 토지의 최근 7일 판 변경사항을 분석해 스탠드업 보고서를 작성해줘. 형식: 완료/진행중/차주 계획.",
  "output": "new_tablet",
```

#### 스킬 마켓 (Phase 3+)

| 항목 | 설명 |
|------|------|
| **공유** | 커스텀 스킬을 은하계에 퍼블리시. |
| **설치** | 다른 사용자의 스킬을 내 모아이에 추가. |
| **유료 스킬** | 크리에이터가 가격 설정. 크리에이터 70% / 판게아 30%. |
| **리뷰/평점** | 설치 수 + 별점 + 리뷰. |
| **권한 투명성** | 스킬 설치 시 요구 권한 표시 (판 읽기/생성/삭제/외부 API 등). |

#### MCP 연동 (Phase 3+)

- 외부 서비스(GitHub, Slack, Figma 등)를 모아이 스킬로 연결.
- MCP Server URL을 등록하면 모아이가 해당 도구를 사용할 수 있음.
- 예: "모아이야, 이 판 내용으로 GitHub 이슈 만들어줘" → MCP를 통해 GitHub API 호출.

### 10.4 모아이의 원칙 (잠금)

1. **모아이는 하나다**. 사용자는 하나의 모아이와 대화한다. 여러 AI가 아니라 하나의 수호신.
2. **스킬은 여럿이다**. 모아이의 능력은 스킬로 확장된다. Phase가 올라갈수록 스킬이 늘어난다.
3. **모아이는 제안하고 사용자는 승인한다**. 자동 변경 기본 금지. diff 미리보기 필수.
4. **스킬은 투명하다**. 어떤 스킬이 사용되었는지, 몇 Ink가 소모되었는지 항상 표시.
5. **잉크가 없으면 잠든다**. Ink 부족 시 모아이는 응답하지 않음 (에러가 아니라 "잠자는 모아이" 연출).
6. **인격은 선택이다**. VTuber 스킨 / 성격 커스텀은 Phase 2+. 기본은 무인격(기능 중심).
7. **프라이버시 우선**. 토지별 AI 토글. 비허용 토지의 판은 모아이에게 전송하지 않음.

### 10.5 모아이 UI

#### 호출 방식

| 방식 | 위치 | 설명 |
|------|------|------|
| **인라인 호출** | 에디터 | 텍스트 선택 → 모아이 버튼 → 요약/번역/톤 변환 등. |

| **측량사 알림** | 자동 (주간) | 측량사 리포트 도착 → 모아이 탭에서 확인 + 승인. |

### 10.6 석공 vs 측량사 — 폐기하지 않고 "스킬 그룹"으로 재정의

석공과 측량사는 사라지지 않는다. **스킬의 카테고리 이름**으로 남는다.

| 이름 | 역할 | 트리거 | 스킬 접두어 |
|------|------|--------|-----------|
| **석공(Mason)** | 사용자가 **요청**할 때 동작하는 스킬 그룹 | 수동 (요청 시) | `mason:*` |
| **측량사(Surveyor)** | **자동/주기적**으로 동작하는 스킬 그룹 | 자동 (주간/이벤트) | `surveyor:*` |
| **장인(Artisan)** (Phase 3+) | **생성/빌드** 계열 스킬 그룹 | 수동 | `code:*`, `asset:*`, `game:*` |

> 세계관에서: 석공은 사용자 옆에서 망치질하는 조각가. 측량사는 대륙을 돌며 지도를 그리는 탐험가. 장인은 Phase 3에서 합류하는 새로운 전문가.

### 10.7 VTuber 스킨 / 인격 (Phase 2+)

- 모아이 위에 선택적 캐릭터 레이어.
- 스킨 = 외형 + 말투. 기능은 동일.
- 스킨 없이 완전 작동 (기본: 🗿 무인격).
- 커뮤니티 스킨 제작/판매 가능 (코스메틱 마켓).

## 10.5) 주민(Resident) — 토지의 캐릭터

### 핵심 요약

- **주민(Resident)** 은 토지에 정착하는 AI 캐릭터. 모아이가 대륙 전체의 시스템 수호신이라면, 주민은 **각 섬의 거주자**.
- 핵심 차별점 — **주민은 그 토지의 기록을 기억한다.** 제타·Character.AI와 달리 사용자의 **실제 판(노트)을 RAG 맥락으로** 대화.
- **모아이 vs 주민**: 모아이=무인격 도구(1개) / 주민=인격 캐릭터(토지별 N개, 선택). 주민은 해당 토지 전체 기록을 지속 기억.
- **주민 2유형**: 실용(코치/멘토/일기 친구 — 일상 토지) / 창작(판타지·롤플레이 — 창작 토지, 세계관/시나리오/비밀 필드 추가).
- **캐릭터 설정 5영역**: 기본(이름/역할/아바타/인사말) + 페르소나(외형/성격/배경/말투/버릇/호불호) + 예시 대화(3~20개) + 관계(유형/친밀도/호칭/만남) + 행동 규칙(always/never/topics).
- **캐릭터 카드**: `.character.json` 표준 포맷. 공유/임포트/Export. Phase 3+ Character Card V2 호환 검토.
- **영입 3경로**: ① 인물도감에서 영입(기본, 30초) ② AI 자동 생성(10~15 Ink) ③ 직접 만들기(크리에이터, 5탭 편집기).
- **인물도감(Codex of Residents)**: Phase 2 런칭. "소수 제작, 다수 소비" 모델. 추천/카테고리/자연어 검색/체험 모드(3 Ink=5회 대화).
- **크리에이터 생태계**: 퍼블리시 조건(필수 필드+예시 대화 5개+소개 100자), 랭크 5단계(새내기→전설), 수익 분배(유료 판매 70/30, 리믹스 원작자 30/리믹서 40/판게아 30, 팁 90/10), 크리에이터 대시보드.
- **감정·관계 시스템 (Phase 3+)**: 친밀도 0~100(낯선→매우 친한), 기분(기쁨/걱정/실망/평온) — 시스템 프롬프트 주입.
- **모더레이션**: 자동 필터(퍼블리시 전/런타임/이미지) + 신고 3회 누적 자동 비공개 + DMCA 48시간 + 성인 콘텐츠 Phase 3+ 별도 섹션.
- **RAG 기억**: 토지 판 벡터 임베딩 → 대화 시 관련 판 Top 5 주입. 판 생성/수정 시 임베딩 갱신(디바운스 5초). 대화 10회마다 요약 → 장기 기억.
- **주민 원칙 9조 (잠금)**: ① 기억은 해당 토지 한정 ② 주민은 기록을 바꾸지 않음(모아이 경유+승인) ③ 주민은 선택 ④ 기억 삭제 가능 ⑤ 주민 간 대화 금지(Phase 2) ⑥ 공유 시 주민 숨김 ⑦ 잉크 소모 표시 ⑧ 영입 캐릭터의 기억은 내 것(원작자 미전송) ⑨ 원작자 크레딧 보존.
- **잉크 소모**: 영입 임베딩 5~10 / 일반 대화 1~3 / RAG 대화 2~4 / 긴 회상 3~6 / 먼저 말 걸기 1 / 체험 3 Ink=5회.
- **핵심 카피**: "제타가 판타지의 캐릭터와 대화하는 것이라면, 판게아는 **당신의 삶을 아는 캐릭터**와 대화하는 것이다."

> 상세 명세(편집기 5탭 UI, 인물도감 화면, 캐릭터 카드 JSON 전문, 친밀도/기분 표)는 원문 v30.9 §10.5.1~10.5.18 참조. Phase 2 구현 시 전문 복원 예정.

- 임포트: Markdown / Obsidian Vault.
- Export: `.md + assets + index.json` (어디서든 열 수 있는 표준 포맷).

### 13.2 동기화 (Phase 1)

- **기본**: 로컬 저장 (오프라인 완전 동작).
- **선택적 중앙 Sync**: 로그인 시 활성화. 기기 간 실시간 동기화.

#### 충돌 해결 정책

| 시나리오 | 해결 | 설명 |
|----------|------|------|
| **동시 편집 (같은 판, 다른 기기)** | 마지막 저장 승리 + 충돌 판 생성 | 늦은 쪽의 버전을 `{판제목}_conflict_{timestamp}.md`로 부두에 저장. 사용자에게 병합 알림. |
| **한쪽 삭제 + 한쪽 수정** | 삭제 보류 + 알림 | 수정된 버전 유지. "이 판이 다른 기기에서 삭제되었습니다. 삭제할까요?" 확인. |
| **양쪽 새 판 (같은 토지)** | 둘 다 유지 | 충돌 아님. 양쪽 모두 정상 동기화. |
| **메타데이터 충돌 (룬 변경)** | 마지막 수정 승리 | front-matter/meta.json의 필드별 타임스탬프 비교. |
| **토지 구조 변경 (이동/삭제)** | 마지막 수정 승리 + 알림 | 판의 `land_id` 변경은 타임스탬프 기준. 충돌 시 부두로 이동. |

#### 충돌 UI

- 충돌 발생 시: 대륙 뷰에서 해당 판 블록에 ⚠️ 뱃지.
- 탭하면: 양쪽 버전 diff 뷰 (좌/우 비교) + "이쪽 선택" / "저쪽 선택" / "둘 다 유지".
- 자동 해결된 충돌도 알림 목록에 기록 (되돌리기 가능).

#### Phase 2 — CRDT 도입 검토

- 실시간 협업(같은 판 동시 편집) 필요 시 CRDT(Yjs 등) 도입.
- Phase 1에서는 동시 편집 불가. "다른 기기에서 편집 중" 잠금 표시.

#### 오프라인→온라인 Sync 큐

| 항목 | 설명 |
|------|------|
| **큐 저장** | 오프라인 중 모든 변경을 `sync_queue.json`에 시간순 기록. |
| **업로드 순서** | 타임스탬프 오름차순(먼저 발생한 변경부터). |
| **충돌 발생 시** | 13.2 충돌 해결 정책 적용. 충돌 판 생성 + ⚠️ 알림. |
| **네트워크 복구 감지** | 온라인 전환 시 자동 업로드 시작. 백그라운드. |
| **진행률** | "동기화 중... (32/100)" 토스트. 완료 시 "동기화 완료" 토스트. |
| **실패 재시도** | 항목별 3회 재시도. 3회 실패 시 해당 항목 건너뛰고 다음 진행. 실패 항목은 "동기화 실패" 알림 + 수동 재시도 버튼. |
| **대용량 큐** | 1000건 이상 시 배치 업로드 (50건씩). UI 블로킹 없음. |

### 13.3 GitHub 연동 — 3단계 모델

GitHub 연동은 백업(레벨 0) → 주 저장소(레벨 A) → 완전 자가 설치(레벨 B)의 3단계로 깊어진다.

#### 레벨 0: GitHub Repo 백업 (Phase 2)

| 기능 | 설명 |
|------|------|
| **GitHub Repo 백업** | 판게아 전체를 GitHub private repo에 자동 push. `.md` + `index.json` 구조 유지. |
| **버전 이력** | Git 커밋 = 판 변경 이력. 별도 버전 관리 불필요. |
| **복원** | Repo에서 pull → 판게아 복원. 기기 분실/초기화 대비. |
| **Obsidian 호환** | GitHub repo를 Obsidian Vault로 직접 열 수 있는 구조. 이중 활용. |
| **GitHub OAuth** | GitHub 로그인도 인증 수단으로 추가. 개발자 타겟. |

#### 레벨 A: GitHub 주 저장소 모드 (Phase 2)

백업이 아니라 **GitHub repo 자체가 데이터의 본체**가 되는 모드. 판게아 중앙 서버에는 데이터가 저장되지 않는다.

```
[판게아 웹앱 (pangaea.app)]
        │ GitHub OAuth (repo scope)
        ▼
[로컬 캐시 (IndexedDB/OPFS)] ←─ 즉시 저장 (로컬 퍼스트 유지)
        │ 동기화 레이어 (5분 배치 or 수동 커밋)
        ▼
[사용자 repo: username/my-pangaea]
   ├─ lands/일터/api-설계.md
   ├─ lands/건강/러닝로그.md
   ├─ assets/{hash}.png
   ├─ residents/코치민수.character.json
   └─ index.json
```

| 항목 | 설계 |
|------|------|
| **저장 흐름** | 편집 → 로컬 캐시 즉시 저장 → Git 동기화는 배치(5분) 또는 수동 "커밋" 버튼. 타이핑마다 커밋하지 않음. |
| **커밋 전략** | Git Data API로 여러 판 변경을 1커밋에 배치. 커밋 메시지 자동 생성("판 3개 수정: API 설계 외"). |
| **충돌 해결** | Git merge 기반. 충돌 시 기존 정책(충돌 판 생성 + ⚠️ 뱃지 + diff 뷰) 그대로 적용. |
| **멀티 디바이스** | 같은 repo를 여러 기기에서 사용. pull → 로컬 캐시 갱신 → push. |
| **Rate limit 대응** | 인증 5,000 req/h. 배치 커밋으로 1인 사용 충분. 초과 시 로컬 캐시로 동작 + 큐 대기. |
| **IStorage 구현** | `GitHubStorage` 어댑터가 `IStorage` 인터페이스 구현. UI/Domain 레이어 변경 없음. |
| **서버 의존 기능** | AI(잉크)·인물도감·공유 SSR·푸시는 판게아 서버 경유 (선택적). |

> 레벨 A 사용자에게 판게아 서버는 "AI와 커뮤니티 게이트웨이"일 뿐, 데이터 보관소가 아니다.

#### 레벨 B: 완전 자가 설치 (Self-Hosted, Phase 3)

**앱 자체를 사용자의 GitHub에 설치**하는 모드. 판게아 서버가 사라져도 영원히 동작하는 "나의 판게아".

```
[템플릿 repo: pangaea-app/template]
        │ ① Fork 또는 "Use this template"
        ▼
[username/my-pangaea-app]  ← 앱 코드 (정적 빌드)
        │ ② GitHub Pages 활성화
        ▼
[https://username.github.io/my-pangaea-app]  ← 나만의 판게아 URL
        │ ③ 데이터 repo 연결 (같은 repo /data 폴더 또는 별도 repo)
        ▼
[완전 독립 동작]
```

##### 설치 플로우 (사용자 관점)

```
1. pangaea.app/self-host 접속 → "내 GitHub에 설치" 버튼
2. GitHub 로그인 → 템플릿 자동 fork
3. GitHub Actions가 자동 빌드 + Pages 배포 (3~5분)
4. username.github.io/my-pangaea 접속 → 온보딩
5. 끝. 이후 모든 데이터는 내 repo에만 존재.
```

##### 기술 설계

| 항목 | 설계 |
|------|------|
| **앱 배포** | 정적 빌드(SPA). GitHub Pages 서빙. 서버 코드 없음. |
| **데이터 저장** | 로컬 캐시(IndexedDB/OPFS) + 같은/별도 repo에 Git 동기화. 레벨 A와 동일 구조. |
| **인증** | ① Fine-grained PAT 직접 입력(기본) ② 판게아 토큰 프록시(선택, 서버리스 함수 1개) |
| **업데이트** | upstream 템플릿 → GitHub Actions가 주기적으로 pull + 재빌드. 사용자는 "업데이트 허용" PR만 머지. |
| **AI (모아이/주민)** | ① BYOK: 사용자가 자기 LLM API 키 입력(설정에 저장, repo에 커밋 안 됨) ② 판게아 Ink 게이트웨이 연결(계정 연동 시) |
| **공유 링크** | 자가 설치 모드에서는 public repo 링크 또는 GitHub Pages 정적 공유 페이지로 대체. SSR 불가. |
| **인물도감** | 판게아 계정 연동 시 사용 가능. 미연동 시 `.character.json` 수동 임포트만. |
| **버전 고정** | 사용자가 특정 버전에 머물 수 있음. 강제 업데이트 없음. |

##### 레벨 B의 의미

| 관점 | 의미 |
|------|------|
| **데이터 소유권** | 앱+데이터 모두 사용자 소유. 판게아가 망해도 영원히 동작. "네 데이터는 네 것"의 완성형. |
| **Phase 5 연결** | P2P 분산 OS 비전의 현실적 첫 형태. TER/온체인보다 먼저 도달 가능한 분산화. |
| **신뢰 자산** | Obsidian 커뮤니티처럼 "탈출구가 열려 있다"는 신뢰가 충성 사용자층을 만든다. |
| **BM 양립** | 자가 설치 사용자도 AI(잉크)·인물도감·공유 SSR은 서버 필요 → 코어 무료 + 서버 기능 수익화 유지. |

##### 모드 비교 요약

| | 중앙 Sync (기본) | 레벨 A: GitHub 주 저장소 | 레벨 B: 자가 설치 |
|---|---|---|---|
| 앱 서빙 | pangaea.app | pangaea.app | username.github.io |
| 데이터 위치 | 판게아 서버 | 사용자 repo | 사용자 repo |
| 판게아 서버 의존 | 전체 | AI/도감/공유만 | AI/도감만 (선택) |
| 버전 이력 | 서버 스냅샷 | Git 커밋 | Git 커밋 |

### 13.3.1 보안 고려 (GitHub 모드 전용)

## 15) 개발 명세(Phase 1 잠금)

### 15.1 ID·파일

- 판 ID: `uuid_v7`/`ulid`. 링크: ID 기반.
- `.md` 판 front-matter: `id`, `title`, `created_at`, `updated_at`, `land_id`, `ward_path?`, `status`, `type`, `priority`, `links[]`, `xset[]`, `tags[]`, `remind_at?`, `pinned?`
- 비-md 판: 동일 필드를 `.meta.json` 사이드카 파일에 저장.
- 디렉토리 구조: `/{land_id}/{ward_path?}/{file}` + `/{land_id}/{ward_path?}/{file}.meta.json`

### 15.2 토지/영역 조작

- 부두 삭제 불가. 토지 삭제 = 기록보관.
- 영역→토지 승격: 판 `land_id` 이동. 핀: `Pinned > Top N > Other`.

### 15.3 교차 정착

- `xset[]`: `{land_id, ward_path?, created_at}`. 원본 삭제 → "끊어진 다리" + 복구.

### 15.4 모아이 스펙

- 석공: 현재 판. 측량사: 7일분 + 부두. 토지 전체 전송 금지.
- diff 미리보기 필수. 실패 → "제안 없음".

### 15.5 주간 리포트(측량사)

1. 부두 분류 제안(토지 배정/룬 추천)
2. 중복 판 3쌍
3. 토지 위생(병합/기록보관/핀)
4. 혼잡 영역 Top 3
5. 다리 추천 Top 5

### 15.6 토템 서버 계약

- 서버 토템 계산 금지. 저장: `land_id`, `codex_id`, `altitude_default`, `monument_placements`.

## 16) MVP 범위 & 개발 태스크

### 16.0 아키텍처 & 디렉토리 구조

#### 레이어 아키텍처

리팩토링과 Phase 확장을 위해 **4레이어 + 인터페이스 경계**를 유지한다.

```
┌──────────────────────────────────────────────────┐
│  UI Layer (View)      — React 컴포넌트. 렌더링만. 로직 금지.    │
│  State Layer          — 앱 전체 상태 관리. Zustand store.       │
│  Domain Layer (Logic) — 비즈니스 로직. 순수 함수. UI 의존 없음.  │
│  Data Layer (Infra)   — 파일 I/O, API 호출, 인덱스, 동기화.     │
└──────────────────────────────────────────────────┘
```

#### 의존성 규칙 (잠금)

```
UI → State → Domain → Data
     (위는 아래에 의존 가능. 아래는 위에 의존 금지.)

✅ UI가 State를 읽는다 / State가 Domain 함수를 호출한다 / Domain이 Data 인터페이스를 호출한다
❌ Data가 UI를 import 하면 안 된다 / Domain이 React를 import 하면 안 된다
```

#### 디렉토리 구조 (원문 기준 — 모노레포 적용 시 packages/core(types·interfaces·domain) + apps/web(app·ui·state·data·styles)으로 매핑)

```
src/
├── app/          # 앱 쉘, 라우팅, 진입점 (App.tsx, Router.tsx, providers/)
├── ui/           # UI Layer — components(Glass/Button/Modal/Toast), continent/(ContinentView, BlockRenderer, IslandLayout, LandLabel, DetailPanel, ZoomPan, LOD, BridgeRenderer), editor/(Editor, MetaPanel, SlashMenu, FloatingToolbar, InlineMoai), ocean/(OceanCanvas, OceanWebGL, OceanInterface★), navigation/, moai/, settings/, onboarding/
├── state/        # Zustand — tabletStore, landStore, uiStore, moaiStore, authStore, syncStore, timeStore
├── domain/       # 순수 로직 — tablet/(activity, frontmatter, search), land/(layout, sort, bridge), totem/(altitude, monolith), economy/(seal, ink, landmass), moai/(router, skills/, types), render/(color, isometric, shadow, time), share/(shareLink)
├── data/         # 인프라 — storage/(fileStorage, metaStorage, draftStorage), sync/(syncEngine, conflictResolver, syncQueue), search/(searchIndex), auth/(googleAuth, appleAuth, emailAuth), api/(llmClient, serverApi), export/(exportPdf, exportZip)
├── interfaces/   # 교체 가능한 인터페이스 ★ — IOceanRenderer, IStorage, ISearchEngine, IAuthProvider, ILLMClient, ISkill
├── __tests__/    # domain 단위 / data 통합 / ui 스냅샷
└── types/        # schema.ts, runes.ts, config.ts
```

#### 핵심 인터페이스 (교체 가능성 보장)

```typescript
// IOceanRenderer — M1 Canvas 2D → M3 WebGL 교체 가능
interface IOceanRenderer {
  init(canvas: HTMLCanvasElement): void;

  writeTablet(id: string, data: Tablet): Promise<void>;
  deleteTablet(id: string): Promise<void>;
  listTablets(landId?: string): Promise<Tablet[]>;
  readLand(id: string): Promise<Land | null>;
  writeLand(id: string, data: Land): Promise<void>;

// ISkill — Phase 3 커스텀 스킬/MCP 확장
interface ISkill {
  id: string;                          // "mason:summarize"

  trigger: 'request' | 'auto';

}
```

#### 상태 관리 전략

- Zustand 기반. 각 도메인별 독립 store. store 간 의존은 subscribe로만. 직접 import 금지.
- Actions: Domain 함수 호출 → Data(storage) → 상태 업데이트.

#### 리팩토링 안전장치: 테스트

| 레이어 | 테스트 종류 | 대상 | 도구 |
|--------|-----------|------|------|
| **Domain** | 단위 테스트 | 활성도 계산, 정렬, 다리 로직, 인장, Landmass | Vitest |
| **Data** | 통합 테스트 | 파일 CRUD, 검색 인덱스, 동기화 | Vitest + mock fs |
| **UI** | 스냅샷 테스트 | 글래스 컴포넌트, 에디터, 상세 패널 | Vitest + Testing Library |
| **E2E** | 시나리오 테스트 | 판 생성→대륙 표시→편집→저장 플로우 | Playwright (M2+) |

### 태스크 작업 규칙

1. **1 태스크 = 1 AI 세션**: 기획서 해당 섹션 참조로 한 세션 구현 가능한 크기.
2. **산출물은 디렉토리 구조를 따른다**: ui/domain/data/state 레이어 구분 필수.
3. **의존성 준수**: 의존 태스크 완료 후 시작.
4. **Domain 레이어는 테스트와 함께**: 완료 시 최소 5개 단위 테스트.
5. **인터페이스 우선**: 교체 가능 모듈은 인터페이스부터 정의.
6. **UI는 State만 읽는다**: React에서 Data 레이어 직접 호출 금지.

## 21) 보안·안정성 지침서

### 21.1 보안 원칙

1. **로컬 퍼스트 = 공격 표면 최소화**.
2. **방어는 계층적으로**.
3. **사용자 데이터는 읽지 않는다**. 서버는 동기화 중개자 (Phase 2+ E2EE 검토).
4. **실패는 안전하게**. 애매하면 차단 쪽으로.
5. **투명하게 공개**. 사고 시 24시간 내 공지.

| 시나리오 | 방지 대책 |

#### 에러 핸들링 원칙

```
1단계: 자동 복구 — 재시도(3회, exponential backoff) → 대체 경로(로컬 캐시)
2단계: 사용자 알림 — 토스트("재시도 중...", "임시 파일에 보존됨")
3단계: 안전한 실패 — 데이터 손실 가능 작업은 실행하지 않음

절대 금지: silent failure / 데이터를 날리는 에러 / 에러 시 빈 화면·무반응
```

### 21.5 보안 개발 지침 (개발자용)

- **코드**: 서버 측 입력 재검증 / 출력 sanitize(DOMPurify) / 비밀은 환경 변수(.env+Vault, 커밋 금지) / 의존성 주간 audit / 로그에 판 내용·비밀번호·토큰 금지 / 스택 트레이스 비노출 / Parameterized queries only.
- **배포**: TLS 1.3+HSTS / CSP(script-src 'self') / CORS pangaea.app만 / DB 매일 백업+분기 복원 테스트 / 모니터링(P0=즉시 알림) / 펜테스트(출시 전·Pro 전·P2 전·연 1회) / API 버전(/v1/, 폐기 6개월 공지).
- **릴리스 체크리스트(14)**: 전 엔드포인트 인증 / 입력 검증 / 인젝션·XSS 테스트 / CSRF / rate limit / 에러 정보 비노출 / 로그 청결 / 의존성 스캔 / redirect_uri / 공유 만료 / 휴지통 30일 / 충돌 무손실 / AI 토글 미전송 확인.

### 21.6 인시던트 대응 절차

```
[감지] → [1] 즉시 격리 (API 비활성화/서버 격리/토큰 무효화)
       → [2] 영향 평가 (30분 내: 사용자 수/데이터 종류/공격 벡터)
       → [3] 사용자 공지 (24시간 내: 인앱+이메일, 범위+조치+가이드)
       → [4] 복구 (패치/세션 초기화/비밀번호 재설정 강제)
       → [5] 사후 분석 (72시간 내: RCA/재발 방지/결과 공개)
```
